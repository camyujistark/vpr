# Ship v0.2 Design Spec

Status: draft
Branch: v0.2
Supersedes: v0.1 `vpr` (kept running until `ship` is feature-complete)

## Problem

v0.1 grouped commits into PRs post-hoc. The core loop was: commit freely → `vpr add` to slice commits into VPRs → write a story → `vpr generate` → `vpr send`. That worked, but three seams kept grinding:

1. **Work decomposition was invisible.** `vpr ticket new` created an Azure WI and a parallel branch, but there was no place for the *plan* — the atomic pieces that make up a PR. todo.md-style lists lived outside the tool.
2. **Post-hoc slicing is the hardest UX moment.** "Which of these 15 commits belong to which VPR?" is a decision that should have been made *before* the commits existed.
3. **Existing Azure backlogs were strangers.** If you already had PBIs and Tasks in Azure, there was no way to pull them into VPR and plan against them — the tool only knew how to *create* new items, not *adopt* existing ones.

## Direction

v0.2 shifts VPR from a post-hoc commit grouper to a **pre-declared PR planner** on top of jj.

| | v0.1 | v0.2 |
|--|------|------|
| Primary unit | commits grouped into VPRs | a plan that declares Epic → Task → PR |
| State store | `meta.json` | `plan.md` (human-readable, canonical) |
| Work decomposition | implicit (commits are the plan) | explicit (atomic tasks listed as commit ranges per PR) |
| Story → Description | per VPR only | at Epic, Task, and PR level |
| Azure backlog | create only | **pull existing WIs**, reference by number, sync descriptions |
| Binary name | `vpr` | `ship` |

The design laws that came out of the v0.2 grill:

1. **Every layer justifies itself against the one-sentence problem.** The problem: "make atomic commits into stacked PRs linked to Azure WIs." If a feature doesn't serve that, it doesn't ship.
2. **Nothing hides state the CLI can't read.** No binary blobs, no TUI-only caches. `plan.md` is truth.
3. **The TUI adds; it never owns, and it never hinders.** TUI is phase 2, implemented as a pure dashboard that shells out to the CLI.
4. **The file is truth. Tools are verbs against the file.** Git tracks history; edits happen in `$EDITOR`; `ship` is stateless between runs.
5. **External tools are shelled out.** jj, gh/az, claude — no vendor lock, no SDKs in the critical path.
6. **Human review gates every AI write.** `$EDITOR` is the review step. No silent overwrites.

## Stack

```
git                    storage, remote, draft history
jj                     stack maintenance on feedback
gh / az                PR creation, WI read/write
claude (or $SHIP_LLM)  LLM shell-out for ship gen
ship (~500 LoC)        this tool
plan.md                canonical plan, one file per working tree
```

Four external tools + one script + one file. No meta.json, no beads, no todo.md separate from `plan.md`.

## plan.md format

Three heading levels. Two labeled fields per section. One commit range per PR.

```markdown
# Epic: MVP1 — PBI 17065

Story:
- first milestone of the ding app
- audio playback for demo
- sandbox mode so we can show without auth
- no real backend yet — mocks throughout

Description:
First milestone of the ding Convertor app. Covers audio playback,
sandbox mode for demo scenarios, and a basic upload flow. Backend
endpoints are mocked via MSW — real integration deferred.

## Task: Audio player — Task 17066

Story:
- adds audio playback to ding
- msw mocks /api/audio/:id
- playwright smoke test

Description:
Adds audio playback to the ding Convertor app. MSW mocks the audio
endpoint since there's no real backend yet. Playwright covers the
golden path.

### PR: AudioPlayer component — targets main

Story:
- core react component with forwardRef
- web audio api for sample-accurate timing
- msw handler for /api/audio/:id
- playwright smoke test

Description:
Adds the AudioPlayer React component with play/pause/seek via Web
Audio API. forwardRef exposes the underlying element. MSW handler
returns a short WAV at /api/audio/:id. Playwright smoke test confirms
the play button fires the expected events.

commits: a1b2c3..e4f5g6

### PR: Wire into ConversionView — targets AudioPlayer component

Story:
- replace placeholder in ConversionView
- thread audio URL from conversion state

Description:
Replaces ConversionView's placeholder with the real AudioPlayer.
Threads audio URL from the conversion state into the component props.

commits: e4f5g6..i7j8k9

## Task: Sandbox mode — Task 17080

Story:
- app runs without auth for demos
- toggle in header, persisted in localStorage
- API calls return canned data when active

Description:
Lets the app run without auth for demos and local exploration. A
toggle in the header persists to localStorage; when active, API
calls return canned data instead of hitting real endpoints.

### PR: Sandbox toggle — targets main

Story:
- toggle in header
- localStorage persistence
- canned responses when active

Description:
Adds a sandbox toggle in the app header, persisted in localStorage.
When sandbox mode is on, API calls short-circuit to canned responses.

commits: x1y2z3..a4b5c6
```

### Heading forms

- **Epic** — `# Epic: <title> — PBI <n>` (or omit `— PBI <n>` if not linked)
- **Task** — `## Task: <title> — Task <n>` (or omit)
- **PR** — `### PR: <title> — targets <base>`
  - `<base>` is either `main`, or `<sibling PR title>`, or omitted (defaults to main for first PR, else previous sibling under the same Task)

### Fields within a section

- `Story:` — human-written. Bullets, prose, fragments, whatever. **Never touched by the tool.**
- `Description:` — `ship gen` writes this from Story + context. Human can edit afterwards.
- `commits: <range>` — H3 (PR) only. Commit range covered by this PR. jj revset syntax accepted.

### Parser rules

- Headings match `^(#{1,3})\s+(Epic|Task|PR):\s+` exactly.
- Field markers match `^(Story|Description):\s*$` at line start.
- `commits:` matches `^commits:\s+(.+)$` — at most one per PR section.
- WI numbers are extracted from heading suffix patterns: ` — PBI <n>`, ` — Task <n>`.
- Targets are extracted from PR heading suffix: ` — targets <name>`.
- Empty fields are allowed. Missing fields are allowed.

## Command surface

```
ship                       push stacked PRs from plan.md
ship gen <level> <name>    polish Description at specified level
ship gen --all             polish every unpolished section
ship plan pull <pbi>       import Azure PBI + children into plan.md
ship sync                  bidirectional Azure sync (descriptions, WIs)
ship status                human-readable overview
ship --help                help
```

### `ship`

Reads `plan.md`. For each H3 (PR):
1. Resolve `commits:` range via jj.
2. Create or update bookmark at last commit of range: `feat/<pbi>-<task-slug>-<pr-slug>`.
3. `jj git push --bookmark <name>`.
4. `gh pr create` (or `az repos pr create`) with:
   - Title: PR heading title
   - Body: Description field + auto-appended `Closes AB#<task-wi>`
   - Base: resolved from `— targets <name>` clause
5. If PR already exists, update title/body if changed; no duplicate create.
6. Idempotent — re-running with same state is a no-op.

### `ship gen <level> <name>`

- `<level>` ∈ `epic`, `task`, `pr`.
- `<name>` is the section title (partial match supported; error if ambiguous).

Behavior:
1. Assemble context for the level (see Context Assembly below).
2. Build prompt template (versioned, lives in `src/prompts/`).
3. Shell out to `claude -p "$prompt"` (or `$SHIP_LLM`).
4. Open result in `$EDITOR` for review.
5. On save: write to `Description` field of the section.
6. On `:q!`: discard, `plan.md` unchanged.

Flags:
- `--all` — gen every section in plan.md with sequential `$EDITOR` sessions.
- `--dry` — print generated text to stdout, don't modify `plan.md`.
- `--pipe` — print the assembled prompt + inputs to stdout, don't call the LLM. For piping to any LLM or inspecting the prompt.
- `--fresh` — ignore current Description when building the prompt; generate from Story only.
- `--yes` — skip `$EDITOR` review (for scripts/CI).

### `ship plan pull <pbi-id>`

Imports an existing Azure PBI and its child Tasks into `plan.md`.

1. Fetch PBI via `az boards work-item show --id <pbi-id>`.
2. Fetch children (typed `Task`) via `az boards work-item query`.
3. Write H1 (Epic) + H2 (Task) sections to `plan.md` with:
   - Titles from Azure
   - `Story:` populated with Azure `System.Description` (as bullets if possible; prose if not)
   - `Description:` left empty (user runs `ship gen` to polish, or edits manually)
   - WI numbers stamped in headings
4. No H3 (PR) sections — those are added when the user plans implementation.

Flags:
- `--append` — add under existing plan.md instead of prompting on collision.
- `--query <wiql>` — provide a custom WIQL query instead of `pbi-id`.

### `ship sync`

Bidirectional description sync between `plan.md` and Azure.

For each H1/H2 section with a WI number:
1. Read local `Description`.
2. Fetch Azure WI description.
3. Compare against last-synced hash (stored in `.ship-sync.json`, gitignored).
4. Four outcomes:
   - **synced** — no diff → skip.
   - **dirty** — local changed → push to Azure.
   - **remote-changed** — Azure changed → pull into local. `$EDITOR` review before apply.
   - **conflict** — both changed → emit merge markers in `$EDITOR`, human resolves.

Flags:
- `--dry` — show what would sync, don't act.
- `--push-only` — skip pull path.
- `--pull-only` — skip push path.

H3 (PR) sections don't sync descriptions at plan time — they sync when `ship` creates/updates the PR body.

### `ship status`

Human-readable view of plan.md:

```
Epic: MVP1 (PBI 17065)                              [synced]
  Task: Audio player (Task 17066)                   [synced]
    PR: AudioPlayer component                       [ready: 3 commits]
    PR: Wire into ConversionView                    [ready: 1 commit]
  Task: Sandbox mode (Task 17080)                   [DIRTY local]
    PR: Sandbox toggle                              [pushed #4953]

Discoveries: 0
Unsynced changes: 1
```

Read-only. No side effects.

## Context assembly per gen level

Each gen level uses different inputs. The generator must produce prose tailored to what that level cares about.

| Level | Inputs |
|-------|--------|
| Epic | Epic Story + `CLAUDE.md`/`README.md` + child Task titles (not bodies) + Azure PBI description if linked |
| Task | Task Story + parent Epic Description + child PR titles (not bodies) + Azure Task description if linked |
| PR | PR Story + parent Task Description + parent Epic Description + `jj diff` for `commits:` range + commit messages in range |

Rationale:
- Epic-level: roll-up framing. Doesn't need child bodies (would echo).
- Task-level: one step of decomposition. Epic gives framing; child PRs give concreteness.
- PR-level: reviewer-oriented. Full context up and the actual diff. This is where the Output most resembles v0.1's output.

Prompts live in `src/prompts/{epic,task,pr}.md` and are templated with `{{story}}`, `{{parent_description}}`, `{{diff}}`, etc. They can be overridden by `.ship/prompts/<level>.md` at repo root for team customization (phase 2+).

## Azure integration

v0.2 includes sync because the user has active backlogs — it's not deferrable.

### What v0.2 supports

- `ship plan pull <pbi>` — one-time import of existing PBI + Tasks.
- `ship sync` — bidirectional description sync, interactive on conflict.
- `ship` — PR creation with `Closes AB#<task-wi>` trailer; Azure Task closes on merge per org config.
- WI numbers referenced in plan.md headings are never silently changed.

### What v0.2 does NOT support

- Auto-creation of Azure WIs when a new Task/PBI appears in plan.md (explicit `ship sync --create` if we need it; not MVP).
- Custom field sync (e.g., effort, priority, iteration). Description only.
- WI state changes from `plan.md` (except auto-close via PR merge, which Azure handles).
- Multiple providers simultaneously (one provider per repo via config).
- Work-item type overrides (phase-2 config). Defaults assume Scrum template: PBI at H1, Task at H2.

### Sync state

Stored in `.ship-sync.json` at repo root, gitignored:

```json
{
  "PBI-17065": {
    "lastSyncedAt": "2026-04-25T10:30:00Z",
    "localHash": "sha256:...",
    "remoteHash": "sha256:..."
  },
  "Task-17066": { ... }
}
```

Regenerable from Azure + plan.md if deleted (forces a full re-sync on next `ship sync`).

## jj integration

jj remains the commit engine. Ship integrates via:

- **Reading commits in a range**: `jj log -r <revset>` for the `commits:` field.
- **Getting diff for gen**: `jj diff -r <revset>`.
- **Creating/updating bookmarks**: `jj bookmark set <name> -r <commit>`.
- **Pushing**: `jj git push --bookmark <name>`.
- **Detecting rebase-induced SHA changes**: ship always resolves `commits:` at run time; it does not cache SHAs.

When feedback comes in:
1. User runs `jj edit <commit>` and amends.
2. jj auto-rebases descendants in the same chain.
3. User re-runs `ship`.
4. Ship resolves the `commits:` range again, gets new SHAs, moves bookmarks, force-pushes.
5. PR bodies stay the same unless `Description` changed.

No separate "restack" command. `ship` is the restack command.

## Migration from v0.1

### What deletes

- `src/commands/add.mjs` — no post-hoc VPR creation; replaced by writing H3 sections in `plan.md`.
- `src/commands/clear.mjs` — no meta.json state to clear; `plan.md` is edited directly.
- `src/commands/edit.mjs` — edits happen in `$EDITOR` against `plan.md`.
- `src/commands/hold.mjs` — hold/unhold moves to a `hold: <reason>` field in `Story:` or just "don't list it in `commits:` yet."
- `src/commands/squash.mjs` — use `jj squash`; ship doesn't wrap it.
- `src/commands/init.mjs` — replaced by `ship plan pull` (existing backlog) or just creating `plan.md` (greenfield).
- `src/commands/remove.mjs` — delete the section in `plan.md`.
- `src/commands/ticket.mjs` — epic/task creation happens via `ship plan pull` or `ship sync --create` (phase 2); otherwise you write sections by hand.
- `src/core/meta.mjs` — `meta.json` is gone.
- `src/core/state.mjs` — state lives in `plan.md` + `.ship-sync.json` + jj.
- Old TUI (`src/tui/`) — replaced by phase-2 dashboard TUI.

### What stays, renamed/adapted

- `src/commands/send.mjs` → `src/commands/ship.mjs`. Simplified: reads `plan.md` instead of `meta.json`, no linearization (H3 `commits:` ranges are already linear per PR), same bookmark-rename + push + PR-create logic.
- `src/commands/generate.mjs` → `src/commands/gen.mjs`. Expanded to three levels (epic/task/pr) with separate context assembly.
- `src/commands/list.mjs` → `src/commands/status.mjs`. JSON output dropped in favor of human-readable; JSON available via `ship status --json`.
- `src/core/jj.mjs` — retained, possibly simplified (no bookmark-rename-during-linearize logic needed).
- `src/providers/*` — retained, used by `ship`, `ship plan pull`, `ship sync`.

### What's new

- `src/commands/gen.mjs` — three-level gen (epic/task/pr).
- `src/commands/plan.mjs` — `ship plan pull`.
- `src/commands/sync.mjs` — `ship sync`.
- `src/core/plan.mjs` — parser + writer for `plan.md`.
- `src/core/sync-state.mjs` — manages `.ship-sync.json`.
- `src/prompts/{epic,task,pr}.md` — templated gen prompts.

### Binary rename

`bin/vpr.mjs` → `bin/ship.mjs`. The `vpr` binary stays for one minor version as a shim that prints a migration message and forwards obvious commands to `ship`.

## Non-goals (explicit)

- Beads integration — concepts absorbed (atomic tasks, story-at-plan-time), tool skipped.
- TUI in MVP — TUI is phase 2, pure dashboard, no generation logic.
- Bidirectional Azure WI *creation* — MVP only syncs existing WI descriptions. Creating new WIs from plan.md deferred.
- Writer/Execution mode split — one CLI, no modes.
- Vim modal mechanics in TUI — not until phase 3 if ever.
- Multi-provider per repo — one provider at a time.
- Rich per-task history / state machines — Git log is the history.
- `.claude/memory/` integration beyond passive reading for project context — Claude Code owns its memory.

## Phased roadmap

### Phase 1 — MVP (this branch)

- `plan.md` parser + writer
- `ship` command (push stacked PRs)
- `ship gen` at epic/task/pr levels with `$EDITOR` review
- `ship plan pull` for existing Azure PBIs
- `ship sync` for description push/pull with conflict markers
- `ship status` for read-only overview
- Azure and GitHub providers
- Spec-aligned tests

### Phase 2 — trigger: you miss it after phase 1

- TUI dashboard (pure viewer + shortcut launcher for gen/ship)
- `.ship/config` for work-item type mapping (Feature/UserStory/etc.)
- `ship plan pull --query` with custom WIQL
- `ship sync --create` to push new WIs up

### Phase 3 — trigger: scale or external pressure

- Richer conflict UI beyond merge markers
- Multi-provider per repo
- Prompt overrides per repo (`.ship/prompts/`)
- Streaming LLM output while it writes
- Integration test harness against a real Azure instance

Each phase is independently shippable. No phase undoes an earlier one.

## End-to-end examples

### Greenfield (no existing backlog)

```
1. Create plan.md by hand with Epic/Task/PR headings and Story bullets.
2. jj new commits for each PR's work. Fill in commits: ranges.
3. ship gen --all      → polish every Description, $EDITOR review.
4. ship                → push stacked PRs, close Tasks on merge.
```

### Existing backlog

```
1. ship plan pull 17065       → imports PBI + Tasks into plan.md.
2. Edit plan.md: add H3 PR sections under each Task with Story bullets.
3. jj new commits for each PR. Fill in commits: ranges.
4. ship gen --all             → polish Descriptions.
5. ship sync --dry            → preview pushes to Azure PBI/Task.
6. ship sync                  → push descriptions to Azure.
7. ship                       → create stacked PRs.
```

### Feedback cycle

```
1. Reviewer leaves comments on PR1.
2. jj edit <commit>; edit; jj describe.
3. jj auto-rebases descendants.
4. ship                       → re-push bookmarks, PRs update.
5. (Optionally) ship gen pr "<title>"  → re-polish Description if diff changed meaningfully.
6. ship                       → push again with updated body.
```

## Open questions for phase-1 build

- Exact commit-range syntax in `commits:` field — jj revset (`a..b`), jj changeId list, or either?
- `ship plan pull` default behavior when `plan.md` exists — append, overwrite with confirm, or refuse?
- `.ship-sync.json` location — repo root or `.ship/sync.json` (nested directory)?
- Behavior when an H3 references a WI that doesn't exist in Azure — warn, fail, or create?

These land during implementation; each has a reasonable default (`a..b`, refuse, `.ship/sync.json`, warn).
