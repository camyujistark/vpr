# VPR — Virtual Pull Request Manager

Gain control over what goes up to git — especially with AI-assisted coding.

AI agents write code fast. Without control, you push up a mess: giant PRs, no narrative, reviewers can't follow. VPR lets you pause, organize work into reviewable PRs, and push clean stacked PRs when you're ready.

## The Problem

You've been coding — maybe with Claude, maybe solo — and you have 30 commits locally. Your team wants small, reviewable PRs that tell a story. You need to:

1. Group commits into logical PRs
2. Write a narrative for each one
3. Push them as a chain where each PR targets the previous
4. Link them to work items

That's what VPR does.

## Concepts

### Items and VPRs

- **Item** — a work item / ticket. The big picture: "Ding Convertor app", "transit.sh improvements"
- **VPR** — a virtual pull request within an item. Each VPR is a small, reviewable chunk with a title, story, and generated output.

```
Item: Ding Convertor app (#17065)
├── VPR: Scaffold              → PR targeting main
├── VPR: Audio components      → PR targeting scaffold
├── VPR: Upload flow           → PR targeting audio
└── VPR: E2E tests             → PR targeting upload
```

### Title, Story, Output

Each VPR has three fields:

- **Title** — the PR title. Short, descriptive.
- **Story** — your narrative. What this PR does, why, what to look for. Written by you.
- **Output** — the PR description body. Generated by AI from the story + the actual commits/diffs. This is what goes on the PR.

The AI takes your story as context, combines it with the work done (commits, files changed, diffs), and produces a concise, templated PR description. You review and edit the output before pushing.

### Parallel Items, Linear VPRs

Items are **parallel branches** off trunk — they don't interfere with each other. VPRs within an item are **linear** — each builds on the previous.

```
trunk
├── ding-app (parallel)
│   ├── ding/scaffold
│   ├── ding/components
│   └── ding/routing
├── transit-sh (parallel)
│   └── transit-sh/idempotent
└── shared-ui (parallel)
    ├── shared-ui/stepflow
    └── shared-ui/media-uploader
```

At push time, VPR linearizes items into a chain, renames bookmarks with work item IDs, and pushes as stacked PRs. Meta is cleaned up after send — fresh slate.

### Source of Truth

**jj owns the graph** — commits, bookmarks, parent-child relationships, conflicts. VPR never fights jj.

**meta.json owns the metadata** — item titles, VPR stories, generated output, work item IDs, event log.

**Bookmark names are the join key** — simple names during development (e.g. `ding/scaffold`), renamed at send time (e.g. `feat/17065-ding-scaffold`).

No indexes during development. No graph mirroring. No renumbering.

## Built on jj

[jj (Jujutsu)](https://martinvonz.github.io/jj/) does the heavy lifting:

- **Parallel branches** — items are siblings, naturally isolated
- **Rebase** — move commits anywhere, jj auto-rebases descendants
- **Bookmarks** — lightweight pointers that follow their commits
- **Conflicts as data** — don't block operations, resolve when ready
- **Colocated mode** — sits alongside git, `jj git push` pushes real git branches

VPR reads jj's graph, attaches metadata, and handles the push ceremony.

## Workflow

### 1. Plan — create items

```bash
vpr ticket new "Ding Convertor app"     # creates work item + parallel branch
vpr ticket new "transit.sh improvements"
```

### 2. Build — make commits freely

```bash
jj new -m "feat(ding): scaffold app"
jj new -m "feat(ding): add AudioPlayer"
# AI agents can commit too — it all stays local
```

### 3. Organize — mark VPR boundaries

```bash
vpr add "Scaffold"                      # mark a VPR within the item
vpr add "Components"
```

Or use the TUI to visually slice commits into VPRs.

### 4. Write the narrative

Each VPR gets a **story** — your intent and context for the reviewer.

```bash
vpr edit scaffold --story "Scaffolds the Ding app with auth, vite config, and smoke test"
```

Or in the TUI: `s` to edit story, `E` to edit all fields in vim.

### 5. Generate output

The AI takes your story + the actual commits/diffs and generates a PR description:

```bash
vpr generate scaffold               # generate output for one VPR
vpr generate --all                   # generate all empty outputs
```

Or use `E` in the TUI — write all stories in vim, save, VPR offers to generate empty outputs, reopens vim with them filled in.

### 6. Move around — jj handles it

```bash
jj rebase -r <commit> -A <target>   # move a commit
jj squash                           # squash into parent
jj split                            # split a commit
```

VPR doesn't interfere. jj moves commits, VPR tracks the metadata.

### 7. Send — push when ready

```bash
vpr send scaffold                   # send one VPR
vpr send --all                      # send all (asks for order first)
```

VPR:
1. Checks story exists, no conflicts
2. Offers to generate empty outputs
3. Linearizes if needed
4. Renames bookmarks with work item IDs
5. Pushes each bookmark as a git branch
6. Creates PRs — each targeting the previous
7. Links work items
8. Cleans meta — sent VPRs move to `sent`

Nothing goes up until you say so.

### Keys

```
j/k         navigate
J/K         scroll right pane
v           toggle diff / file list
Enter       open in $EDITOR (item fields, VPR story, commit message)
n           new item (vpr ticket new)
a           add VPR within focused item
s           edit VPR story
E           edit ALL titles/stories/outputs in $EDITOR
O           set push order in $EDITOR
P           send this VPR
d           dissolve VPR (commits ungrouped)
H           hold/unhold commit
i           interactive rebase in $EDITOR (pick/squash/drop/reword)
u           undo
:           jj command
q           quit
```

### Status indicators

- `✓` sent (pushed, PR created)
- `·` pending
- `!` conflict
- `⏸` held

## CLI

For AI agents and scripting — JSON output where relevant.

```bash
# Items
vpr ticket new "Ding Convertor"     create item + work item + parallel branch
vpr ticket new 17065                attach to existing work item
vpr ticket list                     list all items
vpr ticket edit <name>              update item
vpr ticket done <name>              close item

# VPRs
vpr add "Scaffold"                  create VPR in current item
vpr add "Scaffold" --item ding-app  create VPR in specific item
vpr edit <vpr> --story "..."        write story
vpr edit <vpr> --title "..."        set title
vpr remove <vpr>                    dissolve VPR
vpr list                            JSON: items → VPRs → commits + eventLog
vpr status                          human-readable overview
vpr log [N]                         jj graph with VPR annotations

# AI
vpr generate <vpr>                  generate output from story + commits
vpr generate --all                  generate all empty outputs

# Work
vpr hold <changeId>                 park a commit
vpr unhold <changeId>               release

# Push
vpr send <vpr>                      send one VPR
vpr send --all                      send all (asks for order)
vpr send --dry-run                  preview
```

## Data Model

### meta.json

```json
{
  "items": {
    "ding-app": {
      "wi": 17065,
      "wiTitle": "Ding Convertor app",
      "vprs": {
        "ding/scaffold": {
          "title": "Scaffold",
          "story": "Sets up the app shell with auth, vite, MSW mocks",
          "output": null
        },
        "ding/components": {
          "title": "Components",
          "story": "Ports AudioPlayer, Uploader, ConversionView",
          "output": "## Summary\n- Added AudioPlayer..."
        }
      }
    }
  },
  "hold": [],
  "sent": {
    "feat/17065-ding-scaffold": { "prId": 4952 }
  },
  "eventLog": [
    { "ts": "2026-04-06T...", "actor": "claude", "action": "add", "item": "ding-app", "vpr": "scaffold" }
  ]
}
```

### After send

Bookmarks renamed with work item IDs. VPRs move to `sent`. Items cleaned when all VPRs are sent.

```json
{
  "items": {},
  "sent": {
    "feat/17065-ding-scaffold": { "prId": 4952 },
    "feat/17065-ding-components": { "prId": 4953 }
  }
}
```

## AI Agent Integration

Two interfaces to the same state:

**CLI** — for AI agents. JSON in, JSON out. Agent reads `vpr list`, uses `vpr ticket new`, `vpr add`, `vpr edit`, `jj` for commits.

**TUI** — for humans. Visual overview, vim for editing, preview before pushing.

### Event log

Every action appends to `eventLog`:
```json
{ "ts": "...", "actor": "claude", "action": "add", "item": "ding-app", "vpr": "scaffold" }
{ "ts": "...", "actor": "tui", "action": "squash", "changeId": "abc123" }
```

Claude checks the log to see what the human did. Human sees what Claude did on TUI refresh. Capped at 100 entries.

### Skill file

Tells Claude:
- Read `vpr list` before making changes
- Check `eventLog` for recent human actions
- Use `vpr ticket new` for items, `vpr add` for VPRs
- Write stories for every VPR — the human reviews
- Don't push — the human does `vpr send`

### vpr-send skill

Quality gate before pushing:
1. Review each VPR's commits
2. Verify stories are written
3. Generate missing outputs
4. Check for tests and docs
5. Run tests
6. Preview chain
7. Send one at a time

```bash
npm link
```
